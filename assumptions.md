## Full-bleed hero backgrounds: residual gap accepted at ultra-wide viewports (2026-07-15)

Fixed the "solid black/white gap at hero edges" bug (see progress.md for full detail) by moving each hero's background photo/panel out from under its `max-width`-capped inner wrapper and recentering it on the true viewport with `calc(50% - Npx)`. This makes Services and Contact fully gap-free at any width tested (confirmed to 1920px). Insights, How We Work, About, and Case Studies are dramatically improved but still show a smaller residual gap beyond ~1536-1690px viewport width, because their background images are only moderately oversized relative to the 1440px design and use an exact-pixel-crop-window technique (a specific Figma-designated offset into a larger source image, not a generic center-crop) rather than Services' `object-fit: cover`-any-box approach.

**Deliberately did not** switch these to `width: 100vw` + `object-fit: cover` to close the gap completely, even though that would work mechanically. Reason: `object-fit: cover` crops from the box's center by default, not Figma's specific designated offset — on a box whose aspect ratio no longer matches the source image (which is exactly what happens once the box becomes 100vw-wide on an ultra-wide monitor), this changes which part of the photo shows, risking exactly the "wrong crop vs Figma" bug found and fixed on Services earlier this session (see the "Services hero had the WRONG PHOTO entirely" entry). Flagging as a real trade-off rather than silently shipping either a lingering gap or a silently-drifting crop — revisit only if the user confirms they want full ultra-wide bleed at the cost of some crop drift on those 4 heroes specifically.

## Sitewide 900-1440px gap discovered — fixed for Trusted-By only, rest flagged (2026-07-15)

User reported the Trusted-By logos "overflowing." Root cause was NOT the logos or their bleed positioning (verified no overlap exists at any width via exact `getBoundingClientRect` measurement, at both desktop and mobile) — it's a **sitewide** gap: the site's responsive design only has two tiers (a fixed 1440px desktop layout at >=900px, a separate mobile fallback below 900px), so any viewport in between — common laptop resolutions like 1024px, 1280px, 1366px — has fixed-width content wider than the viewport. Confirmed this affects every single page (21-139 elements per page extend past the viewport at 1024px).

Changed `body`'s `overflow-x` from `hidden` to `auto` — necessary but not sufficient on its own, since 43 of the site's CSS files ALSO set `overflow: hidden` directly on their own section (mostly for legitimate reasons — clipping decorative background patterns, rings, watermark numbers that intentionally bleed past their container), which clips independently of whatever `<body>` allows.

Per explicit user decision, scope was narrowed to just the reported section: `.trusted-by` changed to `overflow-x: auto; overflow-y: hidden;` (preserves whatever vertical clipping was there, only changes horizontal). Verified: at 1024px the section is now genuinely scrollable (`scrollWidth` 1404 vs `clientWidth` 1024) and scrolling reveals the previously-clipped logos (Nolma, Unilever, etc.); at 1440px no scroll room appears (content fits exactly, no unwanted scrollbar); mobile and all other page-level overflow checks unaffected.

**Not fixed, flagged for a future dedicated pass**: the other 42 sections sitewide have the same underlying clipping risk between 900-1440px viewport widths. This was surfaced by the logos being visually obvious when clipped, but the underlying condition (fixed-1440px content only reflowing at <900px) is universal across every section on every page.

## Home hero — 3 real bugs found via native-resolution Figma verification (2026-07-15)

User correctly pushed back that the previous round's "verified, not a bug" conclusion on the doubled hero-outline text was wrong, and asked for genuine re-verification rather than reassurance. Root cause of the earlier mistake: the Figma reference screenshot compared against had been silently downscaled by Figma's own screenshot API (1440px -> 1024px, ~71%), and that downscaling introduced blur on the thin stroke lines that coincidentally resembled the build's actual bug — so two different-looking things were being compared as if equivalent. Re-pulled all references at native resolution (`maxDimension` set above the node's natural size) this time. All 3 reported issues were real:

1. **Doubled/ghosted outline text — real bug, root-caused and fixed sitewide.** At native resolution, Figma's own render is a single clean stroke; the build's `-webkit-text-stroke` was producing a genuine doubled line. Isolated the cause through direct testing (not guessing): tried stroke widths from 1.5px down to 0.15px (no change), the official Google Fonts–hosted file instead of the self-hosted one (no change), an SVG `<text stroke>` instead of CSS (no change — same doubling), and several `-webkit-font-smoothing` values (no change). All of these ruled out stroke-width, font-file quality, and antialiasing as causes. The actual cause: `-webkit-text-stroke`/SVG-stroke both trace the *exact* glyph path, and Montserrat ExtraBold's bold, ribbon-like letterforms (S, e, g, a) have tightly-spaced parallel curves within a single contour — stroking that path directly makes the added stroke lines visually interfere with each other. Figma evidently doesn't stroke the raw path this way. Fix: replaced the technique entirely with an SVG `feMorphology` (dilate) + `feComposite` (operator="out") filter, which dilates the *rasterized* glyph shape and subtracts the original — a morphological "outer silhouette ring" operation that's immune to internal path complexity. Applied via `filter: url(#text-outline-filter)` on the real HTML text (`color: <solid>` instead of `color:transparent`), so the text stays genuinely selectable HTML, not an image — confirmed via `.textContent`. One shared `<filter>` def added once in `src/shell.html` (available on every page); all 7 sitewide instances (Home, About, Services x2, How We Work, Case Studies, Insights, Contact) converted and individually verified with zoomed screenshots.
2. **"Explore Services" button visually close to the meta text above it — hardened, not just patched.** Could not reproduce actual overlap in this environment (measured a real 17px gap, and 166px of spare width before the text would even wrap to a 2nd line — not a razor-thin fit like the Industries card issue). Given the user asked for a fix that prevents overlap "at any point," converted the whole eyebrow → headline → body → meta → actions stack from independent `position:absolute` blocks with hardcoded `top` offsets (fragile: any one block rendering taller than assumed pushes it into the next, fixed-position block) to a normal-flow stack (`.hero__text` wrapper, each child using `margin-top` instead of `top`) — now a taller block always pushes the next one down instead of overlapping it, by construction. Verified pixel-identical positioning to before the change (within 0.2px) at 1440px, and correct stacking/no double-padding on mobile.
3. **Dark strip on the right edge at viewports wider than 1440px — real bug, confirmed and fixed.** Reproduced directly: at a 1920px viewport, `.hero` (no `max-width`, unlike every other section's `__inner` wrapper) stretched to the full 1920px while its absolutely-positioned children (900px photo + 540px cream panel = 1440px) stayed left-anchored, exposing `.hero`'s own `#3a3838` dark background as a ~480px solid strip on the right. Fixed by adding `max-width: var(--page-width); margin: 0 auto;` directly to `.hero` (it has no separate `__inner` layer the way other sections do, so the constraint goes on the section itself). Verified at 1920px: hero now centers correctly (`left:240, right:1680` in a 1920px viewport), no exposed background.

Scanned every other page for the same 3 patterns: no other instance of the doubled-outline bug remained (all 7 sitewide instances share the one shared fix now); no other page showed an actual button/text overlap; no other page's hero showed the dark-edge-strip symptom at 1920px (every other hero uses either a full-bleed cover photo or a proportional 2-column split, both of which scale fluidly — Home's was uniquely a fixed-pixel-width split).

## Navbar + Footer build (2026-07-09)

- **`get_design_context` silently falls back to metadata-only** (no colors/fonts, just positions) for large nodes unless `forceCode: true` is passed. Cost 3 wasted calls before catching this — documented so it isn't repeated.
- **Fonts self-hosted as `.ttf`, not `.woff2`.** Google Fonts' css2 endpoint has a bug/quirk where woff2 requests for this weight range collapse all 5 weights to the same single (Regular) file — confirmed by identical file hashes across weight declarations. The plain TTF fallback path correctly returns 5 distinct files. Converting TTF→WOFF2 locally would need a native build step (`ttf2woff2` requires `node-gyp`/a C++ toolchain), which felt like disproportionate risk for a file-size optimization, so shipping TTF. Flagging in case you'd rather have WOFF2 and can run the conversion in an environment with build tools.
- **Navbar is `position: sticky`, not static.** Figma is a static frame and can't encode scroll behavior; sticky-on-scroll is the standard pattern for this kind of nav and doesn't change the initial-render appearance, so I didn't block on it. Reversible one-line change if you want static instead.
- **Footer's ~407 tiny decorative dot ellipses (4% opacity white, 2.5px, 40×38px grid) implemented as a single CSS `radial-gradient` background pattern**, not 407 individual DOM nodes — verified pixel color/opacity/spacing from the actual Figma SVG asset (`imgEllipse24`) rather than eyeballing. Same visual result, far better for performance/maintainability.
- **Placeholder link targets**: footer social links (LinkedIn/Twitter/Instagram) point to `#` since Figma has no real URLs; "Book a Consultation" CTA links to `/contact.html`. Flag if real social URLs exist.
- Footer Arabic tagline "الحدود تُدفع، النجاح يُقاس" rendered with `direction: rtl` — Montserrat has no Arabic glyphs so it'll fall back to the system font for that line only; this is unavoidable without a separate Arabic webfont (not specified in the design).
- Navbar/Footer both built full-bleed (background spans the viewport) with an inner 1440px-max centered container — Figma frames are exactly 1440px with no wider canvas context, so "what happens beyond 1440px" is inherently a judgment call; full-bleed bg + centered content is the standard pattern for this.
